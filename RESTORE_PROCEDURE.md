# 💾 Резервное копирование и восстановление (Supabase)

Этот проект использует **Supabase** для хранения данных PostgreSQL. Резервное копирование выполняется автоматически.

## 1. Автоматическое копирование
- **Ежедневные бэкапы:** Supabase выполняет полное резервное копирование базы данных каждые 24 часа.
- **PITR (Point-in-Time Recovery):** Если включен тарифный план Pro, доступно восстановление на любой момент времени с точностью до секунды.

### Проверка статуса в Supabase:
1. Перейдите в [Supabase Dashboard](https://supabase.com/dashboard).
2. Выберите проект → **Settings** → **Database**.
3. Проверьте раздел **Backups**. Убедитесь, что PITR включен (рекомендуется 7 дней удержания).

---

## 2. Процедура восстановления (PITR)

Если данные были повреждены или удалены по ошибке, используйте PITR:

1. Перейдите в **Database** → **Backups**.
2. Выберите **Point-in-Time Recovery**.
3. Укажите дату и время (UTC), на которые нужно откатиться.
4. **Внимание:** Supabase создаст **новый проект** с восстановленными данными. Это позволяет проверить данные перед переключением продакшена.
5. После проверки обновите секрет `DATABASE_URL` в Fly.io:
   ```bash
   fly secrets set DATABASE_URL="postgres://postgres:[PASSWORD]@[NEW_HOST]:5432/postgres"
   ```

---

## 3. Ручное резервное копирование

Для создания мгновенного бэкапа перед рискованными операциями:

```bash
# Сохранение бэкапа локально
export PGPASSWORD='ваш_пароль_бд'
pg_dump -h db.[PROJECT_ID].supabase.co -U postgres postgres > backup_$(date +%Y%m%d).sql
```

---

## 4. Еженедельная проверка (Verification)

Для гарантии сохранности данных рекомендуется раз в неделю:
1. Скачать последний бэкап.
2. Развернуть его в локальном Docker-контейнере или временном проекте Supabase.
3. Проверить количество строк в ключевых таблицах:
   ```sql
   SELECT count(*) FROM users;
   SELECT count(*) FROM questions;
   ```
4. Если данные корректны — бэкап считается валидным.
